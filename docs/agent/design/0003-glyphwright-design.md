## 1. Purpose and context

GlyphWright is a turn-based RPG engine in the tradition of RPG Maker and the Neverwinter Nights toolset: game worlds are authored in a grid/tile-based (or room-based) format and played through player/NPC characters, items and inventory, turn-based battles, stats and abilities, and embedded turn-based minigames.

Its distinguishing constraint is that it is **terminal-first and agent-drivable**. GlyphWright serves as a reference application under test for [TermVerify](https://github.com/hoelzl/termverify): an autonomous agent must be able to play, test, and extend GlyphWright games through TermVerify's verification pipeline, using both a direct semantic adapter and real PTY-driven interaction.

GlyphWright is deliberately **not** a TermVerify component. It is an independent application whose design demonstrates what a *well-behaved, verifiable terminal application* looks like — and thereby exercises TermVerify's claim of being harness-neutral (see ADR-001).

## 2. Goals and non-goals

**Goals**

1. A complete, small RPG engine: exploration, NPCs, dialogue, items/inventory, turn-based battle, stats/abilities/perks, turn-based minigames.
2. Two spatial geometries behind one abstraction: grid/tile worlds (presentable as 2/2.5/3D in other frontends) and room-graph worlds (IF-style navigation). A single game may mix both.
3. Determinism as a contract: a run is fully determined by `(engine version, content pack hash, seed, command sequence)`.
4. Semantic evidence first: structured frames and typed events are the primary observations; rendered text is derived.
5. Three frontends over one core: plain line-oriented REPL, JSONL structured stream, full-screen TUI.
6. A stable, versioned public API and wire format that an external adapter (living in TermVerify's examples) can consume without touching engine internals.
7. Content-driven games: worlds, entities, items, abilities, and dialogue defined in data files, validated against schemas.

**Non-goals — permanent**

These contradict what GlyphWright is, and are not expected to be revisited:

- Real-time gameplay. The engine is turn-based to its foundations: turn count *is* time (§5.4).
- A general-purpose scripting language for game logic. Effect primitives are implemented in Python; content composes them.
- Multiplayer.
- Importing `termverify` from engine code, ever (ADR-001). This binds the shipped package; it does not decide whether test or integration tooling in this repository may use TermVerify (§20.5).

**Deferred — out of scope for now, not ruled out**

Nothing in the architecture forecloses these; they are simply not being built yet, and none may bypass the deterministic core when they arrive:

- Graphical rendering. The world model is deliberately presentation-independent, and a graphical frontend consuming the same frames and events *could* be added (§11).
- Animation timing.
- Audio.

Do not cite this section to argue that a deferred item is permanently excluded. Adding one is a scope decision for the owner, recorded as a new design decision — not a refusal.

## 3. Design drivers inherited from TermVerify

TermVerify's design principles translate directly into engine requirements:

| TermVerify principle | GlyphWright consequence |
|---|---|
| Semantic evidence first | `SemanticFrame` + typed event log are canonical observations; ANSI output is derived and covered by few reviewed goldens. |
| Production interaction still matters | The TUI frontend is a real full-screen terminal app driven over a PTY; keybindings map to the same command language the kernel speaks. |
| Determinism is a contract | Pure `step` function, injected seeded RNG, turn-count time, defined iteration order, locale-independent ASCII-first output. |
| Human review owns baselines | Transcripts are human-readable; the plain frontend doubles as the review format. Content-pack and engine fingerprints invalidate stale baselines. |
| Harness-neutral by default | Engine code never imports TermVerify. The introspection meta-channel is a generic engine feature usable from Hermes, Claude Code, CI, or a human at a prompt. |

## 4. Architecture overview

```
                        ┌─────────────────────────────────────┐
  TermVerify PTY   ───► │ Terminal frontends                  │  plain REPL · TUI · JSONL
  (keystrokes,          ├─────────────────────────────────────┤
   screen text)         │ Presentation                        │  SemanticFrame → renderers
  TermVerify direct ──► ├─────────────────────────────────────┤
  (commands, frames,    │ Mode stack                          │  exploration · battle ·
   via glyphwright.api) │                                     │  dialogue · menu · minigames
                        ├─────────────────────────────────────┤
                        │ Deterministic kernel                │  step() · scheduler · events ·
                        │                                     │  seeded RNG · snapshots
                        ├─────────────────────────────────────┤
                        │ World model                         │  Space protocol · entities ·
                        │                                     │  components · content packs
                        └─────────────────────────────────────┘
```

Data flows downward as commands and upward as events and frames. Only the topmost layer knows about terminals; only `glyphwright.api` (§14) is public.

Module layout:

```
glyphwright/
  kernel/        # step(), turn scheduler, event types, rng, snapshots
  world/         # entities, components, Space protocol, grid.py, roomgraph.py, areas/portals
  content/       # pack loading, schema validation, hashing
  modes/         # exploration, battle, dialogue, menu, minigames/
  effects/       # stat pipeline, effect primitives, statuses, abilities, perks
  frames/        # SemanticFrame and view types
  frontends/     # plain.py, jsonl.py, tui/
  harness/       # meta-channel (":"-commands), fingerprints, schema export
  api.py         # the only public programmatic surface
```

## 5. Deterministic kernel

### 5.1 The transition function

The entire simulation is a pure function:

```python
def step(state: WorldState, command: Command, rng: Rng) -> tuple[WorldState, tuple[Event, ...]]
```

No I/O, no wall clock, no ambient globals. All engine behavior — including NPC turns triggered by a player command — happens inside `step`.

Returned event collections are immutable, like state (§5.2): a caller holding a step's events is holding evidence, and evidence that a consumer can append to is not evidence. `step` takes the RNG cursor explicitly even though state carries one (§5.4), so a test can drive a step from an arbitrary point in the stream; the successor state carries the advanced cursor either way.

The command reaching `step` is already valid for the active mode. Validity is answered against the frame's grammar before a step happens, and an invalid command is a typed rejection rather than a step (§6, appendix A.4) — so `step` never needs a "this made no sense" return path.

### 5.2 State

`WorldState` is immutable (frozen dataclasses over structural-sharing maps). Rationale:

- Snapshots are free (`snapshot() = state`), which the direct adapter, save/load, and minigame undo all rely on.
- Tests can assert `step` did not mutate its input.
- Performance is irrelevant at terminal-game scale.

State contains: entity table (id → component bag), area table (id → `Space`), the mode stack, the initiative/scheduler queue, the RNG cursor, the turn counter, quest/world flags, and the message log length (the log itself is derivable from events; only the cursor is state).

### 5.3 Events

Every state change is expressed as a typed event; the successor state is the fold of the event list over the prior state (`apply(state, event) -> state`). Representative event types:

`Moved`, `MoveBlocked`, `ItemAcquired`, `ItemUsed`, `DamageDealt`, `Healed`, `StatusApplied`, `StatusExpired`, `ActorDied`, `ModePushed`, `ModePopped`, `DialogueLine`, `ChoiceOffered`, `FlagSet`, `TurnAdvanced`, `MinigameResolved`.

Events are the engine's semantic evidence. Verification assertions target events ("attacking goblin-1 produced `DamageDealt(target=goblin-1, amount∈[2,5])`") rather than rendered text. Events carry stable entity and position identifiers (§7.5), never screen coordinates.

### 5.4 Time and randomness

Time is the turn counter; there is no wall-clock anywhere in the kernel. Randomness is a PCG64 stream constructed from an explicit seed recorded in the run fingerprint. The RNG cursor is part of `WorldState`, so replay from a snapshot resumes the exact stream. Iteration over entities is in sorted-stable-ID order; dict iteration relies only on insertion order that is itself deterministic from content-pack load order.

The canonical transcript of a run is therefore the triple `(fingerprint, seed, [commands])`, which maps 1:1 onto a TermVerify transcript. Failure shrinking reduces to command-list deletion + replay; replay-from-log doubles as a cheap differential test against snapshot/restore.

### 5.5 Scheduler

A single turn scheduler serves both exploration and battle. After the player's command resolves, the scheduler grants turns to AI-controlled actors (from the initiative queue in battle; from a simple activity list in exploration) until control returns to the player. AI decisions read only world state and the injected RNG, so NPC behavior replays exactly. Battle does not own a private loop; it configures the shared scheduler (ADR-004).

## 6. Command model

Kernel commands are **semantic intents**, not keystrokes:

```
move north            look                  take rusty-key
talk innkeeper        open chest-2          attack goblin-1 with dagger
use potion-minor on self                    cast firebolt at goblin-1
choose 2              equip iron-sword      wait
```

Keystroke handling (arrow keys, hotkeys, menu focus) exists only in the TUI frontend, which translates keys into this same command language. Consequences:

- The direct adapter feeds `Command` objects in-process; no terminal emulation is involved.
- The PTY adapter tests exactly what it should: that bindings, focus, and rendering correctly *reach* the command stream. Differential testing ("same commands ⇒ same frame sequence" across adapters) cleanly separates frontend bugs from simulation bugs.
- Transcripts stay human-readable, supporting reviewed baselines.

**Command enumeration.** Every mode implements `available_commands(state) -> CommandGrammar`: verbs plus argument domains drawn from currently visible/valid referents. The grammar is included in every frame (§11), printable in the plain frontend via `commands`, and gives TermVerify's property/state-machine testing a generator of valid actions at every state — random-walk fuzzing with invariant checks becomes a short test.

Invalid commands are never exceptions; they produce a typed rejection (`Rejected(reason, hint)`) carried in the `StepResult`, so agents receive machine-readable feedback and fuzzing can distinguish "rejected as designed" from "engine fault."

## 7. Spatial model

### 7.1 The Space protocol

Grid worlds and room worlds implement one protocol rather than emulating each other:

```python
class Space(Protocol):
    @property
    def area(self) -> str: ...
    def positions(self) -> Iterable[PosId]: ...
    def exits(self, pos: PosId) -> Mapping[ExitToken, PosId]: ...
    def passable(self, state: WorldState, pos: PosId, mover: EntityId) -> bool: ...
    def blocked_reason(
        self, state: WorldState, pos: PosId, mover: EntityId
    ) -> str | None: ...
    def occupants(self, state: WorldState, pos: PosId) -> tuple[EntityId, ...]: ...
    def observe(self, state: WorldState, observer: EntityId) -> SpatialObservation: ...
```

A space is an **immutable description of terrain and connectivity**; it holds no entities. Occupancy lives in the entity table, so every method whose answer depends on who is currently standing where takes the world state. `positions` and `exits` do not: they are pure topology.

`blocked_reason` returns `None` when the mover may enter, and otherwise names the obstruction in the vocabulary of that geometry — a grid says `wall`, a room graph says `closed`, either says `occupied`. This is what lets a mode report *why* a move failed without knowing which kind of space it is standing in; `passable` is simply `blocked_reason(...) is None`. Without it, modes reach for grid-specific terrain and the abstraction leaks (§7.2).

`move <exit-token>` is the only movement command, everywhere.

### 7.2 GridSpace

`PosId = (x, y)` or `(x, y, layer)`. Exits are derived from adjacency (`north/east/south/west`, optionally diagonals); layer transitions (stairs, ramps, ladders) are explicit exits (`up`, `down`). This is the Neverwinter Nights authoring model: tile-based world description, presentation-independent. `observe` returns a `GridView`: a viewport of tiles around the observer plus visible entities, optionally field-of-view filtered.

### 7.3 RoomGraphSpace

`PosId` is an opaque room ID; exits are authored (`north`, `up`, `through-mirror`). `observe` returns a `RoomView`: room description, contents, exit list — classic IF presentation.

### 7.4 Areas and portals

A world is a set of **areas**, each with its own space kind, connected by **portals** (paired exit entities). One game may combine a grid overworld with room-based interiors; this mixed configuration is the abstraction's acid test and part of the reference content pack.

### 7.5 Position identity

Positions in events, frames, and queries are stable semantic identifiers — `village:7,3`, `cellar:wine-room` — never screen coordinates. Verification assertions must survive rendering changes; identity lives in the world model, not the presentation.

## 8. Entities and content packs

### 8.1 Entity model

Entities are component bags: `id + {Position, Actor, Inventory, Item, Equipment, Portal, Dialogue, Blocker, Renderable, AiBehavior, ...}`. This is component-based design without full-ECS machinery — GlyphWright needs clarity, not archetype iteration throughput. Entity IDs are stable, human-meaningful, and unique per run (`goblin-1`, `rusty-key`).

### 8.2 Content packs

A content pack is a directory of TOML files: species/classes, item and ability definitions, area maps (grid tilemaps or room graphs), dialogue trees, encounter tables, and the world manifest. Packs are validated against published schemas at load time; load failures are hard errors with file/line diagnostics.

A pack hashes to a **pack ID** included in the run fingerprint (§14). Any content change invalidates recorded baselines, so TermVerify can catch "test passed, but against different data."

Game-specific *behavior* enters only through a registry of effect primitives (§9.2) referenced by name from content. There is no embedded scripting language.

## 9. Stats, effects, abilities, perks

### 9.1 Stat pipeline

A stat query resolves through an ordered modifier stack:

```
base value → additive modifiers → multiplicative modifiers → clamps
```

Every modifier carries **provenance**: source (item, status, perk, terrain), duration, and magnitude. The oracle exposes the full derivation (`:query player.stats.atk --explain`), turning "why is attack 17?" from a debugging session into an assertable, self-documenting fact.

### 9.2 Effect primitives and abilities

An **ability** is data: requirements (stats, equipment, resources) + cost + targeting spec (self / single visible actor / exit / tile area) + an ordered list of **effect primitives**. Primitives are the Python-implemented vocabulary — `deal_damage(type, amount_expr)`, `apply_status`, `heal`, `move_target`, `spawn_entity`, `set_flag`, `start_minigame`, `open_dialogue` — composed by content, executed inside `step`, and therefore deterministic and event-producing.

### 9.3 Statuses and perks

A **status** is a bundle of modifiers plus optional event-triggered hooks with a duration in turns. A **perk** is a permanent status acquired through progression. Hooks subscribe to event types ("on `DamageTaken`, if HP < 25%, apply `last-stand`") and fire during the fold inside `step`, so triggered effects appear in the event log like everything else and replay exactly.

## 10. Mode stack

Engine control flow is a pushdown automaton of **modes**. `Exploration` sits at the bottom; `Battle`, `Dialogue`, `Menu/Inventory`, and arbitrary `Minigame` modes push on top. A mode implements exactly:

```python
class Mode(Protocol):
    def available_commands(self, state: WorldState) -> CommandGrammar: ...
    def handle(
        self, state: WorldState, command: Command, rng: Rng
    ) -> tuple[tuple[Event, ...], Rng]: ...
    def view(self, state: WorldState, events: tuple[Event, ...]) -> SemanticFrame: ...
```

`handle` returns the successor RNG cursor alongside its events. A handler that draws (a damage roll, an AI choice) must have its draw land back in world state, or replay would diverge from the original run; returning the cursor is what makes that automatic rather than a rule handlers must remember. Handlers that draw nothing return the cursor unchanged.

`view` takes the turn's events because a frame's `messages` are that turn's log delta, rendered from event data (§11). State carries only the log *cursor*, not the log — the text is derivable from events and nowhere else — so `view(state)` alone could not produce it. A frame requested outside a step (`Engine.frame()`) passes no events and therefore carries no messages, which is correct: nothing just happened.

Mode transitions are themselves events (`ModePushed("battle-3")`, `ModePopped`), hence replayable and assertable.

### 10.1 Battle

Battle is a mode, not a special case. Two presentations are supported by construction:

- **Menu battle** (JRPG style): commands are `attack / skill / item / flee` plus target selection; the frame's viewport is a `MenuView` with combatant summaries.
- **Tactics battle**: the battle mode instantiates a `GridSpace` arena; movement and range reuse the spatial model unchanged.

Initiative lives in the shared scheduler (§5.5). Victory/defeat conditions are checked in the event fold and emit `ModePopped` with an outcome payload consumed by the mode beneath.

### 10.2 Dialogue

Dialogue trees are content-pack data. The dialogue mode emits `DialogueLine` and `ChoiceOffered` events; `choose N` selects. Choices may set flags, grant items, or push other modes (a shopkeeper's `Menu`, a riddle `Minigame`, a `Battle`).

### 10.3 Minigames

A minigame is any mode with its own command vocabulary and frame shape — lockpicking, a card game, a riddle sequence. Because modes inherit the entire kernel/frame/verification machinery, minigames get determinism, replay, enumeration-driven fuzzing, and golden transcripts for free. The reference pack includes one minigame specifically to prove the interface is general (§18).

## 11. Semantic frames

After each step, the active mode produces the canonical observation:

```python
@dataclass(frozen=True)
class SemanticFrame:
    turn: int
    mode: str                        # "exploration" | "battle" | "minigame:lockpick" | ...
    viewport: GridView | RoomView | MenuView | ...
    actors: list[ActorSummary]       # visible actors: id, name, hp, statuses
    messages: list[str]              # this turn's log delta (template-rendered from events)
    prompt: PromptSpec               # what input is expected now
    commands: CommandGrammar         # currently valid commands
```

Frames are pure data. All frontends are pure functions over frames. Message text is generated from templates over event data — never free-written in handlers — keeping prose deterministic and localizable.

Frames and events are what TermVerify compares as primary evidence. Raw glyph/ANSI output is derived material, covered by a small set of reviewed golden snapshots per frontend to catch layout regressions.

## 12. Frontends

**Plain (line REPL).** Renders each frame as an IF-style transcript block — glyph map or room prose, then messages, then prompt — with an unambiguous frame delimiter and turn number. ASCII by default; an optional Unicode tileset sits behind a flag because locale is part of the determinism contract and ASCII transcripts diff cleanly. This frontend doubles as the human review format for baselines.

**JSONL.** Emits the `SemanticFrame` itself, one JSON object per line over stdio, and accepts commands one per line. Lowest-friction transport for agents and for out-of-process verification (§16.3).

**TUI.** Full-screen terminal app (Textual, or hand-rolled ANSI if fewer moving parts under a PTY are preferred — decided at implementation time, see Open Questions) painting stable regions from the same frame: map viewport, status bar, scrolling log, command line. Keybindings translate to kernel commands; nothing else differs. Because plain-text rendering is a pure function of the frame, the differential test "render → parse → equals the rendered projection" is available to verify the plain renderer itself (§17).

## 13. Introspection meta-channel

Alongside game commands, a namespaced meta-channel gated by `--harness` (capability gate, not affiliation — it is equally useful under Hermes, Claude Code, CI, or a human at a prompt):

```
:query player.hp
:query world.entities --at village:7,3
:query player.stats.atk --explain
:events --since 41
:frame --json
:seed
:save slot1        :load slot1
```

Queries use a stable path syntax over world state and never advance the turn — this is the engine's oracle interface. `:save`/`:load` provide setup shortcuts so tests can start mid-dungeon instead of replaying from the title screen. The meta-channel vocabulary contains no TermVerify-shaped assumptions (ADR-001).

## 14. Public API — `glyphwright.api`

The only supported programmatic entry point, versioned independently of internals:

```python
class Engine:
    @classmethod
    def new(cls, pack: ContentPack, seed: int) -> "Engine": ...
    def step(self, command: Command) -> StepResult        # frame + events, or typed rejection
    def frame(self) -> SemanticFrame                      # current frame, no turn advance
    def query(self, path: str) -> QueryResult             # oracle, no turn advance
    def snapshot(self) -> Snapshot                        # opaque, serializable
    @classmethod
    def restore(cls, snap: Snapshot) -> "Engine": ...
    def fingerprint(self) -> RunFingerprint               # engine version, pack hash, seed, turn
```

Everything an external harness needs flows through this surface (command enumeration lives on the frame). Nothing in `kernel/`, `world/`, `modes/`, or `effects/` is public. The producer-side TermVerify adapter — wherever it ends up living (§20.5) — maps this API onto TermVerify's adapter contract; the mapping is expected to be nearly mechanical. **If the adapter needs logic, the API surface is missing something**; that is treated as a GlyphWright bug. The adapter is a consumer of this API like any other, which is what keeps its placement a free choice.

## 15. Wire formats and schemas

With no shared Python types across the verification boundary, the JSONL encodings of frames, events, commands, and query results are the actual interoperability contract:

- Every frame and event carries a schema tag: `"glyphwright.frame/1"`, `"glyphwright.event/1"`.
- Each session begins with a header line carrying the `RunFingerprint` (engine version, pack ID, seed), so TermVerify transcripts record exactly what produced the evidence and baselines invalidate on engine or content change.
- GlyphWright generates JSON Schema files for all wire types from its dataclasses and commits them under `schemas/`. A golden test asserts generated schemas match the committed files, so the contract cannot drift silently. Schema changes are deliberate, reviewed, and versioned by bumping the tag.
- Consumers (the TermVerify-side adapter) validate incoming data against the published schemas. Violations are adapter-level failures, cleanly separating "engine broke its contract" from "game behaved wrongly."

## 16. Verification boundary and packaging

### 16.1 Independence (ADR-001)

GlyphWright is a separate distribution with its own `pyproject.toml` and **no runtime dependency on `termverify`**. The shipped package must be installable and usable with TermVerify absent. The direction is enforced mechanically, not by discipline:

1. An import-linter contract in CI forbids `glyphwright.* → termverify.*`.
2. A dedicated CI job runs GlyphWright's core test suite in a bare environment **without TermVerify installed**, proving independence continuously rather than asserting it in a README.

This constrains the *engine*. It deliberately does not decide where the producer-side adapter lives, nor whether this repository's test/integration tooling may depend on TermVerify — see §20.5.

### 16.2 What lives where

The engine side is settled; the adapter side is not (§20.5).

| GlyphWright package (settled) | Adapter side (placement open) |
|---|---|
| Engine, frontends, content packs | Producer-side adapter over `glyphwright.api` |
| Unit and property tests (plain pytest, no TermVerify) | Adapter conformance suite against the contract |
| Determinism tests: same fingerprint ⇒ same frame sequence | Differential tests: direct vs JSONL-subprocess vs PTY runs of one command sequence yield identical frame sequences |
| Schema-stability goldens | Reviewed golden transcripts (double as living documentation of a well-behaved application under test) |

Wherever the adapter lands, the core suite in the left column must keep passing with TermVerify uninstalled; any TermVerify-dependent tooling is an optional extra, excluded from the bare CI job.

### 16.3 Three adapter flavors

1. **Direct (in-process):** import `glyphwright.api`, feed commands, read frames. Fast properties, replay, shrinking.
2. **JSONL subprocess:** spawn `glyphwright play --frontend jsonl --harness`, speak JSONL over stdio. Process isolation with full semantic observations and no ANSI parsing; composes with sandbox/containment boundaries in a way in-process import cannot. Expected workhorse mode.
3. **PTY:** spawn the TUI (or plain) frontend under a pseudoconsole; drive keystrokes; capture rendered output. Production-interaction evidence.

### 16.4 Version skew

The adapter and engine now drift independently — by design. Schema tags plus a small compatibility matrix in the conformance suite ("adapter N supports frame schemas 1..2") manage it. Exercising this failure mode is itself valuable to TermVerify: real applications under test drift against their harnesses in exactly this way.

## 17. Testing strategy (engine-side)

- **Property/invariant tests** over random valid-command walks (generated from `CommandGrammar`): HP within bounds, inventory conservation, no movement through impassable positions, mode stack never underflows, every event applies cleanly.
- **Determinism tests:** identical `(pack, seed, commands)` ⇒ identical frame and event sequences; replay-from-log equals snapshot/restore.
- **Purity tests:** `step` does not mutate its input state.
- **Schema goldens:** generated JSON Schemas equal committed files.
- **Renderer round-trip:** `parse(render(frame)) == project(frame)`, where `project` names the subset of a frame the plain transcript commits to paper — turn, mode, area, tiles, messages, and the player's HP.

  The round trip is against the *projection*, not the whole frame, and deliberately so. The plain frontend is the human review format for baselines; padding transcripts with the command grammar and every actor's status vector to make them losslessly invertible would wreck the thing that makes them reviewable. Lossless frame transport is the JSONL frontend's job (§12), and its schema goldens cover it. `project` is a declared type rather than an implicit convention, so what a transcript is evidence *of* is itself reviewable, and widening the transcript means widening `project` — a visible change, not an accident.
- **A handful of reviewed golden transcripts** per frontend for layout, updated only by humans.

All engine-side tests are plain pytest, runnable standalone (§16.1).

## 18. Implementation plan

Vertical slices, ordered to exercise the verification boundary as early as possible. The TUI is deliberately fifth: the engine must be fully verifiable before a single ANSI escape exists.

1. **Walking skeleton.** Kernel (`step`, events, RNG, snapshots) + `GridSpace` + `move`/`look` + plain and JSONL frontends + `glyphwright.api` + published schemas. TermVerify side: direct adapter and one PTY golden against the plain frontend.
2. **Items and stats.** Inventory, `take`/`use`/`equip`, the stat pipeline with provenance, `:query --explain`.
3. **Battle.** Menu battle first — forces the mode stack and shared initiative scheduler into existence. Tactics-battle arena as a follow-up reusing `GridSpace`.
4. **Rooms and portals.** `RoomGraphSpace`, plus a portal between a grid area and a room area in the reference pack — the spatial abstraction's acid test.
5. **TUI.** Full-screen frontend; differential tests against plain via the TermVerify-side suite.
6. **Dialogue and one minigame.** Dialogue trees; a lockpicking (or card) minigame proving the mode interface is genuinely general.

Toolchain matches TermVerify to keep the reference consumer honest: Python 3.12+, uv, ruff, mypy (strict), pytest, pre-commit.

## 19. Architecture decision records

**ADR-001 — GlyphWright's engine is import-free of TermVerify.**
*Decision:* Engine code never depends on `termverify`, and the shipped package carries no runtime dependency on it. The adapter consumes only `glyphwright.api` and the published wire schemas — never engine internals. *Scope:* This ADR binds the engine and the distribution. It does **not** decide where the adapter lives or whether this repository's test/integration tooling may use TermVerify; that is deliberately deferred to §20.5. *Rationale:* The stronger test of TermVerify's harness-neutrality claim; keeps GlyphWright a genuinely independent, reusable engine. *Consequences:* The public API and wire formats become versioned contracts (§14–15); engine independence is CI-enforced (§16.1); version skew is managed explicitly (§16.4). *Alternative rejected:* engine natively implementing TermVerify's producer contract — maximally illustrative, but turns the example into a fixture and weakens the neutrality demonstration.

**ADR-002 — Immutable world state with event-fold semantics.**
*Decision:* `WorldState` is immutable; changes are typed events folded into successor states. *Rationale:* free snapshots, trivial replay/undo, testable purity, events as primary semantic evidence. *Consequences:* modest performance cost (irrelevant at this scale); all effect hooks must run inside the fold.

**ADR-003 — Semantic commands; keystrokes exist only in the TUI.**
*Decision:* The kernel speaks intent-level commands; key handling is a TUI-frontend concern. *Rationale:* trivial direct adapter, human-reviewable transcripts, clean differential separation of frontend vs simulation faults.

**ADR-004 — One scheduler for exploration and battle.**
*Decision:* Battle configures the shared turn scheduler rather than owning a private loop. *Rationale:* one determinism story, one replay path, battle as an ordinary mode.

**ADR-005 — Space as a protocol; worlds mix geometries via areas and portals.**
*Decision:* `GridSpace` and `RoomGraphSpace` implement a common protocol; movement is uniformly `move <exit-token>`. *Rationale:* both target genres supported without emulation hacks; mixed worlds become an ordinary configuration and a permanent acid test.

**ADR-006 — Wire formats are versioned contracts generated from code.**
*Decision:* JSON Schemas for frames/events/commands/queries are generated from dataclasses, committed, and golden-tested; instances carry schema tags; sessions carry fingerprints. *Rationale:* no silent drift across a boundary with no shared types.

**ADR-007 — ASCII-first rendering; Unicode tileset behind a flag.**
*Decision:* Default glyphs are ASCII. *Rationale:* locale is part of TermVerify's determinism contract; ASCII transcripts diff cleanly and review easily.

## Appendix A — Wire format sketches

### A.1 Session header (JSONL, first line)

```json
{"schema":"glyphwright.session/1","engine":"glyphwright 0.1.0","pack":"reference-vale@sha256:9f2c…","seed":424242,"harness":true}
```

### A.2 Frame (JSONL)

```json
{"schema":"glyphwright.frame/1","turn":42,"mode":"exploration",
 "viewport":{"kind":"grid","area":"village","origin":[4,0],"tiles":["#####","#..~#","#.@.#","#####"],
             "legend":{"@":"player","#":"wall",".":"floor","~":"water"}},
 "actors":[{"id":"player","name":"Wren","hp":[17,20],"statuses":[]},
           {"id":"goblin-1","name":"Goblin","hp":[6,6],"statuses":[],"at":"village:6,2"}],
 "messages":["A goblin eyes you warily."],
 "prompt":{"kind":"command"},
 "commands":{"verbs":{"move":[["east","north","south","west"]],"attack":[["goblin-1"]],
             "look":[],"talk":[],"wait":[]}}}
```

Every verb maps to a **list of per-argument-position domains**, whatever its arity: `move`
takes one argument drawn from one domain (`[[...]]`), `look` takes none (`[]`). A consumer
walks the same shape for every verb and never special-cases arity — which matters because
this grammar is a fuzzer's generator, and a shape that varies by verb is a shape every
consumer must branch on. Domains are enumerated in sorted order, the same determinism rule
as entity iteration (§5.4), so identical states encode to identical bytes. `south` is a
wall here and is still enumerated: the grammar is topology, not permission (§7.1, A.5).

### A.3 Event (JSONL)

```json
{"schema":"glyphwright.event/1","turn":43,"type":"DamageDealt",
 "source":"player","target":"goblin-1","ability":"strike","damage_type":"slashing","amount":4}
```

### A.4 Rejection

A rejection means *the engine never ran the command*. The turn does not advance and no
events are emitted.

```json
{"schema":"glyphwright.rejection/1","turn":43,"command":"move up",
 "reason":"no_such_exit","hint":"exits here: east, north, south, west"}
```

### A.5 Refusal by the world (not a rejection)

Walking into a wall is a **valid command the world refused**: it is enumerated in the
grammar, it costs the turn, and it produces an event — not a rejection. Compare with A.4.

```json
{"schema":"glyphwright.event/1","turn":43,"type":"MoveBlocked",
 "actor":"player","origin":"village:2,1","exit":"north","reason":"wall"}
```

The distinction is load-bearing:

| | Rejection (A.4) | `MoveBlocked` (A.5) |
|---|---|---|
| Means | the command was never valid here | the command was valid; the world said no |
| In the grammar? | no | yes |
| Turn | not spent | spent |
| Evidence | `StepResult.rejection` | an event in the log |
| Fuzzing reads it as | generator error | engine behaviour |

A grammar advertises the map's *topology*, not today's permissions (§7.1), so `move north`
into a wall is enumerable, attemptable, and answerable with a reason — which is exactly
what makes "attacking a wall tells you it is a wall" assertable. Collapsing the two would
make `MoveBlocked` unreachable and hide real behaviour behind a generator complaint.
